## 2024-04-07 - [Optimization: Using Form XObjects for Grid Patterns in ReportLab]
**Learning:** Found a massive performance bottleneck specific to this codebase's PDF generation architecture: drawing the dot grid background with thousands of individual `c.circle()` operations in nested loops on every single page. ReportLab writes each drawing operator to the PDF stream immediately, which causes excessive processing time (~1.4s just for grids on 10 pages) and massively bloats the PDF file size (~488KB for 10 blank pages with grids).
**Action:** Implemented PDF Form XObjects (`c.beginForm()` and `c.doForm()`) to cache the repeated grid pattern once, and then reuse it across all pages. Combined with using `c.beginPath()` and `c.drawPath()`, this reduced execution time by almost 10x (down to ~0.15s) and dropped the generated PDF file size by nearly 90% (down to ~57KB). When encountering repetitive drawing tasks (like grids or common backgrounds) across many pages in ReportLab, always prefer Form XObjects to caching identical visual components.

## 2024-04-08 - [Optimization: Disabling ASCII Base85 for Images in ReportLab]
**Learning:** Found that ReportLab's default `useA85 = 1` configuration setting forces it to encode image streams using ASCII Base85, resulting in massive CPU overhead (`_py_asciiBase85Encode`). In a typical chapter generation (like chap0), this accounts for >50% of the execution time. Also discovered that text layout functions (`simpleSplit`) are redundantly recalculating the same strings and can be safely memoized.
**Action:** Disabled ASCII Base85 encoding globally via `reportlab.rl_config.useA85 = 0` which drastically improves generation times without significantly hurting final PDF compression (the images are still Flate compressed internally). Additionally, added an LRU cache (`@functools.lru_cache`) over `reportlab.lib.utils.simpleSplit` to speed up text wrapping logic.

## 2024-04-10 - [Optimization: Using Form XObjects for Background Elements]
**Learning:** Extending the XObject optimization (originally just for the dot grid) to cache the background wavy elements and background blobs yielded significant improvements. While generating `test_backgrounds`, repetitive drawing (such as drawing organic path blobs which contain many curve points) on every page causes the number of drawing operators in the PDF stream to stack up quickly. Even complex singular path objects benefit heavily from being cached globally across the document using `c.beginForm()` and `c.doForm()`.
**Action:** When a document uses a common background that has complex paths, waves, or blobs across many pages, encapsulate them in Form XObjects individually or collectively to reduce overhead, generation time, and output PDF size.

## 2024-04-10 - [Optimization: Cached `simpleSplit` text wrapping]
**Learning:** `reportlab.lib.utils.simpleSplit` performs expensive text measurement and wrapping calculations. It is used redundantly in `components.py` functions like `draw_title` and `create_standard_summary_page`. Re-computing splits for the same text and formatting parameters causes measurable CPU overhead across multiple PDF generations.
**Action:** Changed the module-level import in `components.py` from `reportlab.lib.utils.simpleSplit` to use `cached_simpleSplit` from `.utils` (aliased as `simpleSplit`). This seamlessly applied LRU caching to all `simpleSplit` invocations within `components.py`, improving execution time for text-heavy operations by ~18%.

## 2024-04-10 - [Optimization: Globally monkey-patching `simpleSplit` text wrapping]
**Learning:** While aliasing `simpleSplit` locally in modules like `components.py` speeds up direct calls, it fails to optimize `reportlab.platypus.Paragraph` objects, which internally call the original un-cached `reportlab.lib.utils.simpleSplit`. Since Paragraph objects are heavily used for multi-line text blocks (like introductions and summaries), this leaves significant performance on the table.
**Action:** Implemented a global monkey-patch in `Scripts/workbook_generator/utils.py` by reassigning `reportlab.lib.utils.simpleSplit = cached_simpleSplit`. By keeping a reference to the `_original_simpleSplit` inside `utils.py`, infinite recursion is prevented. This ensures both direct text layout operations and ReportLab's internal Paragraph rendering universally benefit from the LRU cache.
