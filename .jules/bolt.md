## 2024-04-07 - [Optimization: Using Form XObjects for Grid Patterns in ReportLab]
**Learning:** Found a massive performance bottleneck specific to this codebase's PDF generation architecture: drawing the dot grid background with thousands of individual `c.circle()` operations in nested loops on every single page. ReportLab writes each drawing operator to the PDF stream immediately, which causes excessive processing time (~1.4s just for grids on 10 pages) and massively bloats the PDF file size (~488KB for 10 blank pages with grids).
**Action:** Implemented PDF Form XObjects (`c.beginForm()` and `c.doForm()`) to cache the repeated grid pattern once, and then reuse it across all pages. Combined with using `c.beginPath()` and `c.drawPath()`, this reduced execution time by almost 10x (down to ~0.15s) and dropped the generated PDF file size by nearly 90% (down to ~57KB). When encountering repetitive drawing tasks (like grids or common backgrounds) across many pages in ReportLab, always prefer Form XObjects to caching identical visual components.

## 2024-04-08 - [Optimization: Disabling ASCII Base85 for Images in ReportLab]
**Learning:** Found that ReportLab's default `useA85 = 1` configuration setting forces it to encode image streams using ASCII Base85, resulting in massive CPU overhead (`_py_asciiBase85Encode`). In a typical chapter generation (like chap0), this accounts for >50% of the execution time. Also discovered that text layout functions (`simpleSplit`) are redundantly recalculating the same strings and can be safely memoized.
**Action:** Disabled ASCII Base85 encoding globally via `reportlab.rl_config.useA85 = 0` which drastically improves generation times without significantly hurting final PDF compression (the images are still Flate compressed internally). Additionally, added an LRU cache (`@functools.lru_cache`) over `reportlab.lib.utils.simpleSplit` to speed up text wrapping logic.
